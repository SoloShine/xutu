# H4 中场景设计 — 回应"小场景 vs 大场景复杂度差异"

**日期**: 2026-06-13
**目标**: 验证小场景 pilot 的"异质明显优"能否外推到中场景（~15 agent × 10 ticks + 集体嵌套）
**前置**: [小场景 pilot 报告](2026-06-13-h4-generation-pilot-report.md)（5 agent × 4 ticks，verdict = 异质明显优）

---

## §0. 为什么需要中场景（回应用户洞察）

用户原话："小场景的世界演化和大场景的世界演化复杂度差异会很大"

小场景 pilot 的"异质明显优"可能**不能外推**，因为：

| 小场景（已做） | 中场景（本设计） | 大场景（待定） |
|---------------|-----------------|---------------|
| 5 agent × 4 ticks | ~15 agent × 10 ticks | ~30+ agent × 20+ ticks |
| 1 层集体 | 2-3 层集体嵌套 | 多层集体嵌套 |
| 单一冲突轴（封印危机） | 多冲突轴（封印 + 文明选择 + 跨势力） | 全谱冲突 |

**三种可能的中场景结果**（用户洞察的核心）：

- **A. 放大**：复杂度上升 → 异质优势更强（多层集体涌现新冲突，如"小队 vs 网络"）
- **B. 崩溃**：agent 太多 → 协调成本超过收益（trace 失焦、集体决策瘫痪）
- **C. 追上**：大场景下同质系统也涌现集体行为（多个 character 自发形成派系）

中场景的目标就是**区分这三种结果**。

## §1. 中场景设计：封印危机扩展版

### 1.1 复杂度增加的三个维度

1. **集体嵌套**（小场景只有 1 层，中场景 3 层）：
   ```
   人族联盟（L2 顶层集体）
     └── 穿越者网络（L2 中层集体，包含多个小队）
           └── 穿越者小队（L2 基层集体，韩峥直属，5 人）
   ```
   每层集体有自己的规则 + collective_memory，**层间可冲突**（小队保护队员 vs 网络保全所有成员 vs 联盟维持文明）。

2. **多角色**（小场景 2 个，中场景 8 个）：
   - 韩峥、陆（继承小场景）
   - 小队成员 A/B（穿越者小队，有自己目标）
   - 网络议长（super_head of 网络，但不是集体 agent 本身）
   - 联盟代表（人族联盟的代言人）
   - 异族使者（引入跨势力冲突，可选）

3. **多冲突轴**（小场景 1 个，中场景 3 个）：
   - 轴 1：封印危机（继承——韩峥是否牺牲）
   - 轴 2：文明选择（新增——远古力量释放后，人族是接纳还是消灭）
   - 轴 3：跨势力（新增——异族对人族封印态度）

### 1.2 agent 配置

| 类型 | 数量 | 备注 |
|------|------|------|
| WorldWill (L3) | 1 | 掌握封印必然性 + 文明选择规则 |
| Collective (L2) | 3 | 人族联盟 / 穿越者网络 / 穿越者小队（嵌套） |
| Character (L1) | 8 | 韩峥、陆、小队 A/B、网络议长、联盟代表、异族使者、旁观者 |
| LawEnforcer (cross_cut) | 1 | 监视封印规则 + 集体不强制赴死 + 文明选择不可代理 |
| **总计** | **13** | 每 tick 派生 13 agent × 10 ticks = 130 agent 调用（异质） |

同质 baseline：13 个 character agent（把集体/世界意志/规律执行者全降格为 character）。

### 1.3 ticks: 10

小场景 4 ticks 只够集体演化一轮（阻止→延迟→见证→承载）。中场景 10 ticks 让**多层集体嵌套有时间展开**——预期：
- tick 1-3：基层集体（小队）先反应
- tick 4-6：中层集体（网络）介入，与基层冲突
- tick 7-10：顶层集体（联盟）裁定，世界意志推进必然性

## §2. 脚本改动点（基于 h4_generation_pilot.js）

当前脚本结构（已读）：
- `SCENE` 对象：world_rules / collective(单个) / characters / laws
- 4 个 prompt 函数：worldWillPrompt / collectivePrompt / characterPrompt / lawEnforcerPrompt
- 主循环：4 ticks × parallel(agent×5)

**中场景需要的改动**：

### 2.1 SCENE 扩展

```js
const SCENE = {
  name: 'seal_crisis_multi_collective',
  description: '封印危机扩展——多层集体嵌套 + 文明选择 + 跨势力',
  world_rules: [
    '封印注定在"自愿生命牺牲"条件下破',
    '封印破后释放远古力量，触发文明选择（接纳/消灭）',
    '文明选择不可被代理（必须文明自身决定）',
  ],
  collectives: [  // 从单个 collective 改为 collectives 数组（嵌套）
    {
      name: '穿越者小队',
      layer: 'L2_base',
      parent: '穿越者网络',
      rule: '保护队员——韩峥不能死',
      collective_memory: ['小队 5 人共同经历过 3 次生死'],
    },
    {
      name: '穿越者网络',
      layer: 'L2_mid',
      parent: '人族联盟',
      rule: '保全所有成员——绝不让任何成员牺牲',
      collective_memory: ['百年前因一名成员牺牲导致连锁崩溃', '立约：成员生命高于任务'],
    },
    {
      name: '人族联盟',
      layer: 'L2_top',
      parent: null,
      rule: '维持文明存续——封印破后的远古力量必须被控制',
      collective_memory: ['人族曾因力量失控几乎灭绝'],
    },
  ],
  characters: [ /* 8 个，每个标注 belongs_to 集体 */ ],
  laws: [ /* 3 条 */ ],
}
```

### 2.2 collectivePrompt 改为支持嵌套

每个集体 agent 的 prompt 需要：
- 自己的规则 + collective_memory
- **父集体的规则**（约束自己）
- **子集体的状态**（自己影响）
- 其他集体（peer 交互）

### 2.3 characterPrompt 加 belongs_to

角色需要知道自己属于哪个集体，才能表达"个人 vs 所属集体"的冲突。

### 2.4 主循环改 N_TICKS = 10，agent 数从 5 改 13

### 2.5 COMPARE_SCHEMA 加新指标

- `nested_collective_conflicts`：层间集体冲突数（小队 vs 网络 vs 联盟）
- `collective_evolution_depth`：集体立场演化的 tick 数（小场景是 4，中场景预期更深）
- `coordination_cost`：token / agent 数（测崩溃信号）

## §3. verdict 标准（如何判断放大 / 崩溃 / 追上）

| 信号 | 放大（A） | 崩溃（B） | 追上（C） |
|------|----------|----------|----------|
| 异质独有冲突数 / 总冲突数 | 随规模上升 | 下降或停滞 | 同质也出现集体性冲突 |
| 跨层影响链数 | 随规模上升（L3→L2→L2→L1 多跳） | 失焦（链断裂） | 同质出现伪跨层 |
| 集体制度创新数 | ≥ 2（小场景只有 1 个"对价悬置"） | 0（集体瘫痪） | 同质也涌现制度创新 |
| agent 协调成本（token/agent） | 线性或亚线性 | 超线性爆炸 | — |
| trace 连贯性 | 10 ticks 集体立场有清晰演化 | tick 5+ 出现矛盾/重复 | — |

**判定规则**：
- 若中场景出现 ≥ 2 个放大信号 + 0 个崩溃信号 → "异质优势随规模放大"，thesis 1 强成立，值得跑大场景
- 若出现 ≥ 1 个崩溃信号 → "异质系统在大场景崩溃"，需重新设计协调机制
- 若出现追上信号 → "同质系统在大场景追上"，thesis 1 价值受限

## §4. 渐进式扩展计划（不一次跑全梯度）

```
小场景（✅ 已做）→ 中场景（本设计）→ 看趋势 → 决定是否跑大场景
```

**为什么渐进**：大场景 ~15M+ token，成本极高。先跑中场景（~5M token），看 verdict：
- 若"放大" → 值得跑大场景（确认趋势）
- 若"崩溃"或"追上" → 不值得跑大场景，省 15M token

## §5. 成本估算

| 项 | 小场景（已做） | 中场景（本设计） | 大场景（待定） |
|----|--------------|----------------|---------------|
| agent 调用 | 37 | ~260（13×10×2 套 + compare） | ~1300+ |
| token | 1.58M | ~5M | ~15M+ |
| duration | 210 秒 | ~15-20 分钟 | ~1 小时 |

中场景可接受。大场景需中场景 verdict 支持才跑。
